Maven安装指南|Maven安装教程
2014-03-29 ⁄ 工具使用 

1:maven是一种项目管理工具，比ant以及make的功能要强大，是项目管理与构建的自动化工具，对程序员而言更多的是偏向于自动化构建功能。

2：maven的安装

在安装maven之前必须要保证的是安装了JDK，因为当运行maven此会对JDK进行检查，JDK的地址就不说了，maven可以从官网上进行下载。

将从官网上下载下来的Maven解压，此处我放在D:\apache-maven-3.0.5 ,为了使用maven的bin目录下的可执行文件，必须将bin目录加入Window环境变量中，配置如下：

Maven官网：http://maven.apache.org/download.cgi

3:安装完后，可以在cmd.exe中输入如下命令，检测其是否安装完成mvn –v或mvn –version 如果出现下图界面，则证明配置成功

至于为什么会出现上述图三的结果，可以看看bin/mvn.bat¸这是windows的批处理文件，当然在linux下应该看的是mvn这个文件，其为shell脚本，从mvn.bat中可以发现如下代码

@REM ==== START VALIDATION ====
if not "%JAVA_HOME%" == "" goto OkJHome

echo.
echo ERROR: JAVA_HOME not found in your environment.
echo Please set the JAVA_HOME variable in your environment to match the
echo location of your Java installation
echo.
goto error

:OkJHome
if exist "%JAVA_HOME%\bin\java.exe" goto chkMHome

echo.
echo ERROR: JAVA_HOME is set to an invalid directory.
echo JAVA_HOME = "%JAVA_HOME%"
echo Please set the JAVA_HOME variable in your environment to match the
echo location of your Java installation
echo.
goto error
:chkMHome
if not "%M2_HOME%"=="" goto valMHome

if "%OS%"=="Windows_NT" SET "M2_HOME=%~dp0.."
if "%OS%"=="WINNT" SET "M2_HOME=%~dp0.."
if not "%M2_HOME%"=="" goto valMHome

echo.
echo ERROR: M2_HOME not found in your environment.
echo Please set the M2_HOME variable in your environment to match the
echo location of the Maven installation
echo.
goto error

看到了吧，里面对环境变量JAVA_HOME以及M2_HOME进行了检测,根据JAVA_HOME检测是否安装了JDK,因为其要运行java –v来判断JDK的版本，M2_HOME就不用说了，所在运行Maven前必须要有JDK

Maven建立HelloWorld程序

HelloWorld程序是程序员入门的第一个程序，同样来使用Maven建立第一个项目HelloWorld

在第一次执行这条命令时,必须要保证联网，因为Maven需要从网上下载一些jar包以及一些插件，jar包与插件的保存位置，我们可以从conf/settings.xml文件里得到，如下代码：

localRepository
   | The path to the local repository maven will use to store artifacts.
   |
   | Default: ~/.m2/repository
  <localRepository>/path/to/local/repo</localRepository>

从这里可以看到默认的保存目录为~/.m2/repository，其中~为用户的根目录，在Windows下即为用户的个人文档。

命令运行的结果如下图

在执行的过程中会让用户进行选择，均选择默认类型即可

Archetype可以理解为此项目的项目类型

-Dname对项目的名字以及包名进行定义

-DgroupId一般可以取为公司名或域名com.test
-DartifactId用来定义项目的名称
-Dversion 用来说明项目的当前版本，当发布的时候我们可以看到相应的jar包后会有

执行完上述命令后，可以到D:\Users\pengpeng.pp\workspace\HelloWorld目录下看看，可以发现

如下目录结构：

目录src/main中是程序的源文件，test目录中放的是测试文件，在根目录下有一个pom.xml文件是整个项目的配置文件，对项目进行描述，插件配置，以及依赖关系的建立.以下是工程使用的pom.xml文件，如下：

<project xmlns="http://maven.apache.org/POM/4.0.0" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 http://maven.apache.org/xsd/maven-4.0.0.xsd">
  <modelVersion>4.0.0</modelVersion> 
  <groupId>pp</groupId> 
  <artifactId>HelloWorld</artifactId> 
  <version>1.0</version> 
  <packaging>jar</packaging> 
  <name>HelloWorld</name> 
  <url>http://maven.apache.org</url> 
<properties>
  <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding> 
  </properties>
<dependencies>
<dependency>
  <groupId>junit</groupId> 
  <artifactId>junit</artifactId> 
  <version>3.8.1</version> 
  <scope>test</scope> 
  </dependency>
  </dependencies>
  </project>

如上所示：在POM文件中，groupId, artifactId, packaging, version 叫作 maven 坐标，它能唯一的确定一个项目,有了 maven 坐标，我们就可以用它来指定我们的项目所依赖的其他项目，插件，或者父项目,如上所述，该项目又依赖于另一个maven坐标,其项目录名为junit. 上面列出的POM文件并不是运行时的POM文件，运行时的POM文件要远比这个POM文件复杂的多，这是因为每一个POM文件都是继承自SUPER-POM，不然怎么会有那么多默认目录呢,要想看到运行时的POM可以使用如下命令：

mvn help:effective-pom

最后说明一点POM文件中的scope说明依赖关系的适用范围，即junit 的 scope 是 test，那么它只会在执行 compiler:testCompile and surefire:test 目标的时候才会被加到 classpath 中，在执行 compiler:compile 目标时是拿不到 junit 的。

还可以指定 scope 为 provided，意思是 JDK 或者容器会提供所需的jar文件。比如说在做web应用开发的时候，我们在编译的时候需要 servlet API jar 文件，但是在打包的时候不需要把这个 jar 文件打在 WAR 中，因为servlet容器或者应用服务器会提供的，scope 的默认值是 compile，即任何时候都会被包含在 classpath 中，在打包的时候也会被包括进去。

对整个项目文件进行编译，编译的命令如下如示:

mvn compile

编译之后的项目文件会多了一个目录HelloWorld/target,在这个目录中会有编译生成的字节码文件App.class

执行以下命令

mvn test
